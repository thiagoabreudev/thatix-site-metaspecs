## Tipos de Mudança

- **major**: Breaking change (altera significado, remove campos obrigatórios, muda estrutura incompatível)
- **minor**: Adição compatível (novas seções, novos campos opcionais, exemplos adicionais)
- **patch**: Correções e clarificações (typos, links, formatação)
- **baseline**: Primeira versão versionada

---

## Princípios de Versionamento

### Quando Incrementar MAJOR (x.0.0)
- Mudança de significado de termos chave (ex: "limite" passa a significar algo diferente)
- Remoção de seções obrigatórias
- Alteração incompatível de estrutura
- Mudança de personas principais
- Alteração de precificação que quebra compatibilidade

### Quando Incrementar MINOR (0.x.0)
- Nova seção adicionada
- Novos campos opcionais
- Nova persona adicionada (sem remover existentes)
- Novos exemplos ou casos de uso
- Expansão de conteúdo compatível

### Quando Incrementar PATCH (0.0.x)
- Correção de typos
- Clarificações de texto
- Atualização de links
- Ajustes de formatação
- Correções de datas

---

## Processo de Atualização de Versão

1. **Identificar tipo de mudança** (major, minor, patch)
2. **Atualizar `spec_version` no frontmatter** da spec
3. **Atualizar `last_updated`** para data atual
4. **Se MAJOR**: atualizar `supersedes` para versão anterior
5. **Documentar mudança** no bloco `:::breaking_changes` da spec
6. **Registrar mudança** neste arquivo (VERSION_HISTORY.md)
7. **Atualizar versão no índice** correspondente (business/index.md ou technical/index.md)
8. **Comunicar mudança** para stakeholders (se relevante)

---

## Context Governance

Este sistema de versionamento garante:

✅ **Context Drift Prevention**: Mudanças são rastreadas e visíveis
✅ **Semantic Regression Detection**: Breaking changes são identificáveis
✅ **Context Clash Avoidance**: Versões incompatíveis não se misturam
✅ **Auditoria Completa**: Histórico de evolução do contexto
✅ **Rollback Capability**: Possível voltar para versão anterior se necessário
